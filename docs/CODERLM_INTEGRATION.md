# GDC 확장 전략: CodeRLM 접근법 흡수

> 상태: Proposed  
> 작성일: 2026-02-12

## 1) 목적

GDC와 CodeRLM 비교 논의를 바탕으로, GDC의 확장 방향을 명확히 정의한다.
핵심 목표는 다음과 같다.

- GDC의 `spec-first` 정체성 유지
- LLM 구현 정확도를 높이는 코드 인덱싱 기능 보강
- 운영 복잡도와 유지보수 비용 최소화

## 2) 의사결정 요약

결론은 다음 세 가지 옵션 중 **"GDC 중심의 느슨한 결합(loose coupling)"** 을 채택한다.

1. **완전 통합(단일 제품화)**: 채택하지 않음
2. **완전 독립 사용(수동 병행)**: 채택하지 않음
3. **GDC 중심 확장 + CodeRLM 아이디어 선별 흡수**: 채택

## 3) 판단 근거

### 3.1 완전 통합을 채택하지 않는 이유

- **스택 이질성**: GDC(Go)와 CodeRLM 서버(Rust)를 강결합하면 배포/운영 복잡도가 급증한다.
- **권위 모델 충돌**: GDC는 `spec-first`, CodeRLM은 `code-first` 성격이 강하다. 단일 권위가 모호해질 수 있다.
- **제품 단순성 훼손**: GDC의 강점인 CLI 중심 단순 워크플로우가 약화될 수 있다.

### 3.2 완전 독립 사용을 채택하지 않는 이유

- 사용자(또는 에이전트)가 두 도구 출력을 수동으로 합쳐야 하므로 운영 비용이 높다.
- 프롬프트 구성 단계에서 반복적인 컨텍스트 스위칭이 발생한다.

## 4) 확장 원칙

1. **YAML 스펙이 권위(Source of Truth)**  
   설계 의도, 책임, 계약은 스펙이 기준이다.

2. **코드 인덱스는 보조 증거(Evidence)**  
   구현 현실(실제 코드/호출자/테스트)은 스펙을 보강하는 용도로 사용한다.

3. **기본 동작은 서버 비의존(Zero mandatory daemon)**  
   기존 GDC CLI만으로 현재 기능이 동작해야 한다.

4. **옵션 기반 점진 도입(Opt-in)**  
   신규 컨텍스트는 `extract` 옵션으로 켜고 끌 수 있어야 한다.

5. **역호환 유지(Backward compatible)**  
   기존 `gdc init/sync/check/extract` 워크플로우를 깨지 않는다.

## 5) 구현 로드맵

### Phase 1: 코드 파싱 정확도 향상 (우선순위: 높음)

목표: `gdc sync --direction code`의 신뢰도 개선

- 비Go 언어(C#/TypeScript) regex 파서를 tree-sitter 기반으로 단계적 교체
- Go는 기존 `go/ast` 강점을 유지하고 필요 시 보완
- 파싱 결과의 스펙 병합 정책(설명 보존, 계약 해시 갱신) 명확화

산출물:

- 파서 정확도 회귀 테스트
- 파서 실패/부분 파싱 케이스에 대한 진단 메시지 개선

### Phase 2: `gdc extract` 컨텍스트 확장 (우선순위: 높음)

목표: 설계 의도 + 구현 현실을 한 프롬프트로 결합

제안 옵션:

- `--with-impl`: 대상 노드의 현재 구현 코드 스니펫 포함
- `--with-tests`: 관련 테스트 파일/테스트 함수 후보 포함
- `--with-callers`: 주요 호출자(역참조) 정보 포함

원칙:

- 기본 출력은 기존과 동일
- 신규 옵션은 명시적으로 활성화할 때만 포함

### Phase 3: 코드 탐색 명령 확장 (우선순위: 중간)

목표: CLI 내 탐색 생산성 향상

- `gdc search <pattern>`: 코드 검색 래퍼(예: ripgrep 연계)
- `gdc trace --reverse <node>`: 역참조 추적 강화
- 심볼/구현/호출자 조회를 위한 쿼리 명령군 설계 (`gdc query ...`)

## 6) 기대 효과

- GDC의 강점(설계 의도 전달, 아키텍처 검증)을 유지한 채 실코드 근거를 보강
- AI 프롬프트 품질 향상: 계약 중심 + 구현 현실 동시 제공
- 기존 사용자 워크플로우를 깨지 않는 점진적 확장

## 7) 리스크 및 대응

- **리스크: 파서 복잡도 증가**  
  대응: 언어별 단계 도입, 회귀 테스트, 실패 시 기존 파서 fallback 정책 정의

- **리스크: 프롬프트 과대화**  
  대응: `extract` 옵션을 기본 OFF로 유지하고 토큰 예산 가이드 제공

- **리스크: 스펙/코드 불일치 증가**  
  대응: `gdc check`에 불일치 진단 규칙을 추가하고 sync 리포트 강화

## 8) 최종 결정 문장

GDC는 CodeRLM을 제품 단위로 병합하지 않는다. 대신, CodeRLM의 핵심 아이디어(정확한 코드 인덱싱/타겟 조회)를 GDC에 점진적으로 흡수한다. 이때 **스펙은 권위, 코드는 보조 증거**라는 원칙을 유지한다.
